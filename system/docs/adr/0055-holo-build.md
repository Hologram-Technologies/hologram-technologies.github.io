# ADR-0055: Holo Build — the third native holospace pillar: a serverless, VS Code-faithful IDE where every object's code is content-addressed, edits re-render live, and humans + AI agents co-edit converging to one κ

**Status:** Accepted — witnessed: `tools/holo-build-witness.mjs` is green (9/9 checks in real Chromium) and `#holo-build` is a required row in `os/etc/conformance.jsonld` (the gate, ADR-024); re-run by `tools/gate.mjs` (23/23). The app ships at `Hologram Apps/apps/workspace/index.html` (κ-pinned in its `holospace.lock.json`, boots 0-fallbacks in the OS frame, audit-apps green). Composes Holo Forge (ADR-0051, the build-as-κ-transform + the O(1) build memo), build·run·share (ADR-0049/holo-app), the durable content-addressed κ-store + read/write substrate (ADR read-write), and the SDK (ADR-0050). Reuses the real VS Code editor (Monaco) and terminal (xterm.js), κ-verified like any other holospace byte (Law L5).

**Context.** A holospace already had two of the three verbs every creative act needs: **Run** (a κ executes — ADR-0049/0051/0052) and **Share** (the κ *is* the share — holo://κ, self-verifying, serverless). The missing pillar was **Build** — the place a human or an agent actually *writes and shapes* the code that becomes those κ's. The developer's home for that act is VS Code, and the instinct "just run VS Code in the browser" has two honest answers, both wrong for us: code-server / openvscode-server run the *whole* product but require a **server** and a Node extension host (not serverless); vscode.dev runs only a *web-extension subset* behind a service worker. Neither is UOR-native, and neither makes the holospace's own objects — its files, functions, builds, receipts — first-class, editable, content-addressed things. The deeper point: VS Code's *editor* is already free and best-in-class (Monaco is `src/vs/editor` extracted verbatim), and its *workbench* is a layout + a set of design tokens we can reproduce faithfully. What VS Code cannot give us serverlessly — a universal extension host — the substrate replaces with something stronger: extensibility as **verifiable content-addressed objects** (skills/MCP, ADR-035/049) and **WASM language servers** (a language server is just another κ-object, like the Forge compiler). So Build is not "port an IDE"; it is "make the holospace's content-addressed graph editable through the most familiar surface in software, with no server."

**Decision.** Ship **Holo Build** as the `workspace` holospace: a VS Code-faithful IDE that is 100% serverless and content-addressed end to end. Concretely:

1. **Adopt VS Code's real components, verbatim and κ-verified.** The editor is Monaco (the extracted VS Code editor — multi-cursor, find/replace, folding, bracket-matching, IntelliSense/hover/go-to-def via the documented `monaco.languages` API, minimap); the terminal is xterm.js. Both are vendored, content-addressed, and verified by re-derivation on load (Law L5) — nothing is trusted from a bare `<script src>`. The "Dark Modern" look is VS Code's own token hex (from `microsoft/vscode` theme-defaults) fed to `monaco.editor.defineTheme` — the same component, faithfully themed (ADR-0029: adopt the pattern, not a foreign runtime).

2. **Reproduce the workbench shell** (not in Monaco): activity bar, resizable side bar (Explorer + Search), **multi-file editor tabs** with per-file models (own undo/cursor) + dirty state, breadcrumbs, the panel (Problems · Terminal · Object Graph · Run), a faithful status bar, **Quick Open (Ctrl+P)** and **Command Palette (F1 / Ctrl+Shift+P)** — a few KB of HTML/CSS around the real components, zero new dependencies.

3. **Content-addressing AS performance (O(1)).** Per the hologram thesis — "performance is content-addressing, not micro-optimization; a memo hit is O(1)" — every distinct content is hashed once and the κ reused everywhere (`shaCache`); one compile is shared by diagnostics · run · graph (`compileCached`); an identical Holo Forge build **rebinds in O(1)** (the durable build memo — rebind, not recompute, surfaced as "⚡ O(1) rebind"); auto-save updates only the active file's κ in place, never re-hashing the tree; collaboration broadcasts are suppressed when solo — so text input stays crisp with near-zero per-keystroke work.

4. **The reflective, live Build.** Every object is content: a file is a κ (Explorer + the status-bar κ), a function and its calls are a κ-graph (the Object Graph panel — nodes are functions, edges are calls, click to jump), a build artifact + its PROV-O receipt are κ's (Holo Forge). The IDE is a *window onto that graph*; editing the source re-derives it. A holospace's running code renders **live** in the Run panel — Holo-C compiles + executes; an HTML holospace renders in a sandboxed, content-addressed iframe that hot-reloads on every edit. Edit → re-derive → re-render, instantly.

5. **Real-time human ⊕ agent collaboration that converges by κ.** A zero-server `BroadcastChannel` carries cursors, selections and edits between every peer in the holospace (panes/tabs + the in-browser agent); presence shows as avatars (status bar, like Live Share) and remote carets with name flags. Crucially, peers don't merely merge — they **converge to the same content address (κ)**, so agreement is *verifiable by re-derivation* (Law L5), not trusted CRDT bookkeeping. An agent is a first-class peer via `window.HoloCoedit` (`join · setCursor · applyEdit · peers`), and acts on the substrate via MCP build·run·share (ADR-0049).

6. **Web2 ⊕ Web3 ⊕ AI, one surface.** It just runs in a tab (web2); everything is content-addressed, offline-first in IndexedDB, shareable as holo://κ with no backend (web3); and it is built for agents — MCP verbs, the co-edit API, structured compile errors for fix-and-retry loops (AI).

A required `#holo-build` row + `holo-build-witness.mjs` prove all of this offline in real Chromium: the editor + terminal mount serverlessly; an identical build rebinds in O(1); files open as tabs; a holospace renders live; Quick Open + Command Palette work; and two peers co-edit and **converge to one κ**.

**Consequences.** Build · Run · Share are now all native holospace verbs over one content-addressed substrate, all keyed off κ: you shape code in a familiar IDE, it runs by κ, and the κ *is* the share — no server at any step, online or off. The honest cost is the boundary we will not cross to keep that promise:

*Non-goal — universal VS Code extension compatibility.* Running *all* VS Code extensions requires a Node extension host and a server (code-server) or, at best, the web-extension subset behind heavy infrastructure (vscode.dev). That is fundamentally incompatible with "100% serverless, runs offline on any device," so we do **not** claim it. The substrate's answer is better aligned and more powerful: (a) **language intelligence via WASM language servers** — an LSP/DAP server compiled to WebAssembly is a content-addressed κ-object, verified and run in-tab, giving rich IntelliSense/diagnostics/format/debug for any language with no server (the natural ADR-0056); (b) a **web-extension subset host** (Open VSX `browser`-entry extensions in a Web Worker) as a later, optional phase; (c) **holo-native extensions as verifiable κ-objects** — a skill/MCP tool (ADR-035/036/049) is discoverable, content-addressed, agent-usable, and conscience-gated (ADR-033), which is a stronger extensibility model than trusting a marketplace blob. Other deferred surfaces (split editor groups, a settings UI, a full SCM view — reimagined as **κ-history**, since every save is a κ and any prior κ can be diffed/restored without git, and DAP debugging) are additive and do not change the architecture.

What is unlocked *because* the substrate is content-addressable, beyond VS Code: **κ-history / time-travel** (every edit is a content address — local history and "restore any version" with no git, no server); **verifiable, reproducible builds** in the editor (Forge receipts, ADR-0051); **collaboration whose agreement is provable** (κ-convergence, not trust); **offline-first by construction** (the whole project is κ-objects in IndexedDB); and **one editable graph spanning web2/web3/AI** addressed uniformly. Build is the on-ramp: the place humans and agents jointly author the objects that the rest of the stack runs, owns (ADR-0053), settles (ADR-0048) and shares.
