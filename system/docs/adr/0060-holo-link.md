# ADR-0060: Holo Link — content-addressed composition: ship a linker, not a compiler. Every module is a self-verifying WASM κ-object that imports others by content address, so objects combine into new objects and split apart, connected only by one self-referential κ

**Status:** Accepted — witnessed: `tools/holo-link-witness.mjs` is green (9/9, pure Node, in the gate's LIVE set) and `#holo-link` is a required row in `os/etc/conformance.jsonld` (the gate, ADR-024); `tools/gate.mjs` green. Also verified in the real browser IDE (`window.HoloApp.run` links a κ-import in-tab; the κ's match the Node witness byte-for-byte). The compiler gained a `extern … from "κ";` declaration and emits a WASM import section (`os/usr/lib/holo/holo-forge/holo-forge.mjs`); the linker lives in `holo-app.mjs` (`run` → `linkGraph`). Composes Holo Forge (ADR-0051) and build·run·share (ADR-0049); ships inside Holo Build (ADR-0055).

**Context.** The honest escalation of Holo Forge from a toy compiler to the real C/C++/Rust universe pointed at LLVM — but a reproducible clang+sysroot in WebAssembly is ~100 MB, fights determinism, and is anything but lean. The first-principles question is not "how do we run a compiler in a tab" but "what already gives us a universal, content-addressable compute substrate that's *already in every browser?*" The answer is **WebAssembly itself**. WASM is the universal IR every toolchain lowers to; it runs natively, serverless, fast, in every engine; and a `.wasm` module is just bytes — a κ-object. So the leverage was never *compilation*. It is **composition**. The gap in the stack was tiny and exact: `run(κ)` instantiated modules with an empty imports object, so a module could not name another. Fill that — let a module's imports be *other κ-objects* — and the κ-store becomes a universe of interoperable, composable, serverless WASM apps. Ship a linker (hundreds of lines), not a compiler (hundreds of megabytes).

**Decision.** Make WASM imports resolvable by content address, end to end:

1. **The link is the κ.** A module names a dependency by putting the provider's **κ in the WASM import's module-name** (`(import "did:holo:sha256:…" "add" (func)))`). The κ is therefore *in the module's own bytes*, so the module's identity (its κ) transitively commits to its dependencies — the self-referential link the substrate runs on. Holo-C gains one declaration — `extern int add(int a,int b) from "did:holo:…";` — and the compiler emits the import section **only when externs are present**, so every existing program compiles to byte-identical wasm (the additive change moved only the compiler's own κ + the receipt that embeds it; all fixture artifacts re-derive to their pins).

2. **The linker is content-addressed, verified, and O(1).** `run(κ)` walks the module's declared imports (`WebAssembly.Module.imports`); each import whose module-name is a κ is resolved from the store, **verified by re-derivation (Law L5)**, and instantiated — **once per graph** (memoized → a shared dependency links exactly once, Law L3) — then wired export→import. A non-κ import can still be satisfied by the caller (the root's own imports). The whole composition is reached from one κ; no manifest, no registry, no bundler, no server.

3. **Closure.** The set of κ-objects is closed under composition: combine modules and you get a κ-object; split one out and each part is still its own self-verifying, independently runnable κ-object. The witness proves the full loop — a library builds to a κ; an app imports it by κ (its bytes commit to the dependency's κ); running links + executes correctly across the boundary (Law L5 semantic); the library still runs standalone; recomposition is O(1); the composition shares as one `holo://κ`; and a **tampered dependency refuses the entire run**.

This turns Build·Run·Share into **Build·Compose·Run·Share**. In the IDE: build a library → κ; paste that κ into another file's `extern`; Run links the graph in-tab; Share is the composition's κ.

**Consequences.** It deletes the software-distribution stack rather than optimizing it. Package management dissolves — a dependency is a κ you reference (no registry to trust, no install, no version conflicts, no diamond problem; two "versions" are two distinct objects), resolved + verified + cached on first run and O(1) forever. Deploy dissolves — build → κ → share → it runs in any browser, verified, offline; shipping *is* sharing an address. A shared component (a stdlib, a UI kit, a codec) is built once *anywhere* and reused everywhere by content (the network gets faster as it grows, Law L3). Supply-chain trust dissolves into re-derivation: every app is a Merkle-DAG of κ-objects with PROV-O receipts; a forged dependency cannot reproduce the pinned κ, so it is refused. And because an app is now an object, the rest of the stack applies to it unchanged — a composition can be **owned** (Title, ADR-0053), **settled per proven use** (ADR-0048), **delegated** (UCAN, ADR-0042), **conscience-gated at mount** (ADR-0033), and **composed by agents** over MCP whose work is a re-derivable DAG (ADR-0045). Polyglot interop comes for free at the WASM layer: a Rust, Go, or Zig module compiled *anywhere* is just another κ-object the substrate composes — we host none of their toolchains.

**Rich, typed data composes too — WIT-style, isolated** (the canonical-ABI core). WASM core functions pass only numbers, so passing a *string* across a κ-link needs the bytes to move between memories. The faithful answer — the one the **WASM Component Model / WIT** standardizes — is that **each component keeps its OWN linear memory** and the *linker lifts and lowers* values across the boundary. We ship exactly that, minimally: a component that uses memory **defines + exports its own** memory (Holo-C gains `load`/`store`/`load8`/`store8` over it), an `extern` parameter may be typed `str` (lowering to `(ptr,len)`), and the string-type info travels in a content-addressed **`holo-iface` custom section** — so a module's own bytes (its κ) commit to its interface *types*, not just its dependency addresses. At link time, when component A calls a `str`-typed import, the linker reads the bytes from **A's** memory, allocates space in **B's** memory (via B's exported allocator), copies them in, and rewrites the pointer — the canonical-ABI lift/lower. The witness proves both the crossing and the **isolation**: app A builds `"ABC"` in its own memory and calls `bytesum(str)` → 198, while calling B's `bytesum` directly on the same address returns 0 (A's data is *not* in B's memory — it was copied, not shared).

The honest edge: this slice carries **strings** (and any byte-buffer) across isolated components, plus all scalars. What remains are the richer WIT shapes — string/record **return** values (the indirect-return convention), and `record`/`list`/`variant` lowering — each an additive increment over this same custom-section + lift/lower mechanism, no change to the resolution model; and it stays lean because the browser already runs component-model modules. The closure property is fully real today; what widens is the *vocabulary* of the typed interfaces. This is the LLVM vision's result — universal, composable, verifiable, serverless compute — with none of its weight, because WASM is the universal IR and the substrate ships only the address-resolving linker.
