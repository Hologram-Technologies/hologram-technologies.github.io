# ADR-0084: Holo Q Mixture-of-Specialists — the helper tasks stop defaulting to one big model and each auto-discovers the BEST small specialist the open web offers, run on-device and proven by receipt: "auto" stops meaning "use the main model" and starts meaning "the right tiny mind for this exact job"

**Status:** Accepted (selection + routing landed + Node-witnessed + validated against the live Hugging Face Hub; the FIRST execution adapter — Session search / embeddings — landed + proven end-to-end; remaining provider adapters + shell wiring + `#holo-q-mux` row pending). The Mixture-of-Specialists layer ships at `system/os/usr/lib/holo/q/holo-q-mux.js` (buildless, dependency-free, DOM-free). The first adapter, `holo-q-embed.js`, takes the **Session search** task all the way: it auto-discovers bge-small, runs it on-device through the OS's already-vendored transformers.js (auto WebGPU/WASM, ORT-in-worker, q8 weights), binds it as the task's provider, and seals a re-derivable embedding receipt — with a content-addressed cache (identical text → O(1)), live telemetry (self-optimizing), and a `reevaluate()` swap-recommendation hook the orchestrator ratifies (self-improving). Proven through the real runtime (`Xenova/bge-small-en-v1.5`, q8): 384-dim, **≈6 ms/embed**, reproducible (same text → identical vector), correct semantic recall (an on-topic doc 0.73 vs off-topic ~0.35, ranks #1); `autowire()` falls back to the deterministic reference embedder so recall never blocks. Node-witnessed 8/8 (cache · receipt · autowire+fallback · recall ranking · reevaluate keep/swap). The layer itself maps the nine helper tasks (the model-router surface — `Vision · Web extract · Compression · Session search · Skills hub · Approval · MCP · Title gen · Curator`) each to a Hugging Face pipeline + a size ceiling, discovers candidates with one cheap serverless API call per task, ranks them with a PURE deterministic score, and routes each task through a per-task provider registry that falls back to the main model when nothing browser-runnable fits. Node-witnessed 13/13 (URL · runnability gate · size parse · deterministic ranking · pick · honest fallback · registry · autoAssign · describe); validated live — the real Hub surfaces sensible tiny picks (`BAAI/bge-small-en-v1.5` for recall, `Falconsai/text_summarization` for page summary, `distilbert-…-sst-2` for approval). **Deferred:** the REMAINING per-task provider adapters (classifier · generative · vlm — the embedding adapter is done), each loading the chosen ONNX/GGUF as a κ-disk on the engine seam its job needs; the shell wiring that makes each "Change / Set to main" row live; a hard size cap via a detail fetch; and a required `#holo-q-mux` witness row + os-closure re-lock. Composes Holo Q (ADR-0052), QVAC (ADR-0067), Holo Mind (ADR-0081, the orchestrator), the hologram-ai ONNX→`.holo` compiler (ADR-0017), the Constitution (ADR-0033), and the decode-agnostic receipt (ADR-0024/0025/0083).

**Context.** Hologram's helper tasks — the small, frequent jobs around the edges of a session (summarize this page, embed for recall, classify an approval, title a chat) — all default to the one main model ("auto · use main model"). That is wasteful and often *worse*: a 100M–500M model fine-tuned for summarization or sentence-embedding beats a general model at its one job, at a fraction of the latency, and the open web already hosts thousands of them. The substrate already has every primitive to exploit this — a model is content-addressed bytes resolved as a κ-disk (ADR-0052), the QVAC contract takes a pluggable per-call brain (`useBrain()`, ADR-0067), and the native orchestrator can run a discover→select→act→verify loop (ADR-0081). What was missing is the glue that turns "auto" from a fallback into an *intelligence*: a layer that, for each task, finds the best tiny specialist, runs it in the tab, and proves the result — with the user never touching a model picker. The danger is over-promising: most Hub weights cannot run in a browser, and downloading a candidate to evaluate it would destroy both "fast" and "serverless." So the design's discipline is to keep discovery cheap and selection pure, and to be honest when no specialist fits.

**Decision.** Ship a **Mixture-of-Specialists** layer the orchestrator drives, factored so the cheap/serverless parts and the heavy/honest parts are cleanly split:

1. **Discover — one cheap serverless call per task.** Each helper task declares a Hugging Face `pipeline_tag` and a parameter ceiling. `discover()` issues a single Hub API query (`/api/models?pipeline_tag=…&sort=downloads`) — a browser `fetch`, no server, no auth — and never fetches a candidate's weights to judge it.
2. **Select — a PURE, re-derivable ranking.** `rankCandidates()` scores the returned metadata only: a hard gate on **browser-runnability** (an `onnx` / `transformers.js` / `gguf` marker present — others are unrunnable in a tab and dropped), then pipeline match, a size-fit term (within the cap, smaller is better; over the cap is disqualifying), popularity (log downloads + likes), and an open-license nudge. Ties break by id, so the same Hub response always yields the same pick (Law L5). The chosen specialist is a **plan**, not a loaded model.
3. **Bind — the plan becomes a κ-disk.** Loading streams the chosen model's weights as a content-addressed κ-disk (every sector re-derived before use, ADR-0052), adapts it to the engine seam its job needs (embedding · classifier · generative · vlm), and registers it in a **per-task provider registry** (`bindSpecialist(task, provider)` / `routeTask(task)`). This is the QVAC `useBrain()` idiom widened from one active brain to one-per-task.
4. **Verify + fall back, honestly.** Each task output seals the SAME decode-agnostic, re-derivable InferenceReceipt and passes the fail-closed conscience gate (ADR-0033/0083). When no browser-runnable specialist fits, or there is no WebGPU, `routeTask()` returns the main model — it **never blocks and never fakes** (Law L5 voice, twin of `holo-q-diffusion.js`'s honest-pending path).

**The magic is the simplicity.** The router surface stays *exactly* as plain as it is today — a list of tasks, each saying "auto." Nothing new to learn, no model picker. But "auto" is now quietly genius: behind a subtle cue (the specialist's name · size · source) one tap reveals the *why* and the receipt. Every hard part — discovery, the runnability gate, ranking, κ-disk verification, capability adaptation, caching, routing, fallback — is hidden. Abstract all complexity; reveal one calm surface. This is the orchestrator (ADR-0081) earning its keep on the most ordinary work in the OS.

**Honest costs.** Only browser-runnable models are eligible — the open Hub is large but the in-tab-executable slice is the real catalog, and that filter is load-bearing. Size is *estimated* from the model id/tags (a hard cap needs a per-pick detail fetch, deferred — so an unlabeled large model can currently slip past on popularity). The remaining work is the rest of **per-task provider construction**: the **embedding** adapter (Session search, `holo-q-embed.js`) is done and proven (≈6 ms/embed, correct recall); the **classifier** (Approval · MCP · Curator), **generative** (Web extract · Compression · Title gen), and **vlm** (Vision) adapters follow the same shape but are not yet built. Tiny specialists vary in quality, so the main-model fallback is not just for "no WebGPU" but a quality floor the orchestrator can choose. Cold-start latency on first use is hidden behind the κ-cache and the fallback, not eliminated. Shell wiring (making each router row live) + a required `#holo-q-mux` witness row + the os-closure re-lock are owed.

**Composition.** Extends 0052 (Holo Q) and 0067 (QVAC `useBrain()`), driven by 0081 (Holo Mind's discover→select→act→verify loop); executes via 0017 (hologram-ai ONNX→`.holo`) and the κ-disk; verifies via 0024/0025/0083 (the decode-agnostic receipt) and 0033 (conscience). Sibling to 0083 (Holo Q Diffusion) under the one inference transform. Mints nothing — the task→pipeline map and the ranking are data.
