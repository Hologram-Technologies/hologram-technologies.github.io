# ADR-0083: Holo Q Diffusion — a masked-diffusion LM is a SECOND decode family under the one inference transform: the receipt is decode-agnostic, so block-by-block denoising re-derives exactly as token-by-token greedy does

**Status:** Accepted (seam landed + Node-witnessed; WebGPU forward kernel + `#holo-q-diffusion` row + κ-re-lock pending). The deterministic control plane ships at `system/os/usr/lib/holo/q/holo-q-diffusion.js` (buildless, dependency-free, DOM-free), bound behind the existing `HoloQVAC.useBrain()` contract — no new façade, no new receipt shape. The QVAC spec (`holo-qvac.mjs`) gains a `DECODE` family list and a `DREAM_V0_INSTRUCT_7B` model descriptor (`decode:"diffusion"`, `provisioned:false`). Verified in Node (6/6): the schedule is `linspace(1,eps,steps+1)`; `decode()` re-derives byte-identically across two passes; a provider with no kernel **throws an honest pending error rather than faking tokens** (Law L5); `describeDiffusion()` carries the validated provenance. **Deferred:** the WebGPU masked-diffusion forward kernel (the only device-specific piece), the int4→κ-disk publication pass, a required `#holo-q-diffusion` conformance row + witness, and the os-closure κ re-lock (the whole `qvac` group is unsealed on this branch). Builds on **Holo Q** (ADR-0052, verifiable inference) and **Holo QVAC** (ADR-0067, the encoded SDK contract); composes Forge (0051), the UOR envelope (0025), PROV-O realizations (0024), and the Constitution (0033).

**Context.** Holo Q proved inference is a re-derivable κ-transform on one fact: *greedy decode is deterministic*, so the GPU path mirrors the CPU executor token-for-token and the same prompt+model+params yields identical tokens everywhere. That made the receipt — `κ(prompt ⊕ context ⊕ model ⊕ params ⊕ engine) → κ(output)` — a re-derivable observation rather than an authority's claim. But it baked in a hidden assumption: that decoding is **autoregressive**. The whole engine (the WebGPU ternary brain + the hologram-ai CPU floor's `generate_stream`) produces one token per sequential forward pass, left to right. A **diffusion** language model breaks that shape: it holds the whole output block masked and FINALIZES it by iterative, bidirectional UN-masking over a fixed denoising schedule — many tokens per step, in confidence order, not position order. Dream-7B (`Dream-org/Dream-v0-Instruct-7B`, a Qwen2.5-7B backbone re-trained to denoise) is the concrete instance. The question this ADR answers: does a diffusion brain VIOLATE the receipt model, or generalize it?

**The finding (measured, not assumed).** It generalizes it. With a **fixed step schedule + confidence-ranked unmasking + temperature 0**, diffusion decode is fully deterministic. Validated empirically: Dream-7B, quantized to int4 (group-64 RTN) and run on CPU (`Models/Dream/holo_dream_q4_stream.py`), produced a coherent answer ("a hologram is a three-dimensional image … appears to float in the air") and — run twice — the **byte-identical output κ** (`2a67997cb7b1725d`). So a diffusion run is a re-derivable observation by the same Law L5 argument as greedy AR. It is arguably *cleaner*: the number of forward passes is fixed by `steps`, independent of the output length, so the cost (and the proof obligation) is bounded up front. The receipt is **decode-agnostic** — it already commits to `params`, and the step schedule is just more params; a verifier re-runs the same schedule on the same model and reproduces the output-tokens κ. Nothing in the receipt, the conscience gate, or the O(1) memo changes.

**Decision.** Ship the diffusion brain as a **second decode family**, not a second engine, with the heavy and the deterministic parts cleanly split:

1. **The deterministic control plane lives in the substrate, re-derivable.** `holo-q-diffusion.js` implements the entire diffusion control loop in pure JS: the `linspace(1,eps,steps+1)` schedule, the per-step count of positions to finalize (`round(masked·(1−s/t))`), the confidence ranking (max-softmax, ties broken by position → pure), and greedy token choice. This is the part that must re-derive, so it runs identically in Node and the browser (the Holo Atlas isomorphism). `decode()` is a pure generator: `(promptIds, genLen, steps, eps, kernel) → finalized block`.

2. **Only the forward pass is delegated to a device kernel.** The one device-specific, matmul-heavy piece — the 7B forward producing per-position logits — is an injected `kernel.logits(tokenIds)`. The WebGPU masked-diffusion kernel that implements it is the deferred work (and the natural home for the per-step parallelism that makes diffusion fast on a local GPU). Splitting it out means the proof surface (the schedule + selection) is small and auditable, while the unverifiable-by-eye part (the matmul) is content-addressed by the model κ-disk.

3. **Honest until bound — never a faked output.** Without a kernel, the provider's `generate()` THROWS a structured "pending (ADR-0083)" error, which `completion()` surfaces as a `completionError`. It never streams a fabricated answer — the same Law-L5 voice as the façade's `unprovisioned()` and the dormant `useHoloQ()` seam. Dream is registered `provisioned:false`; the reference floor still answers generic completion. `describeDiffusion()` reports exactly what is proven (determinism, the int4 recipe) versus pending (the kernel).

**What this enables.** A faster local brain (a fixed handful of parallel denoising steps vs. one sequential pass per token — the biggest practical win for serverless browser-GPU inference); native fill-in-the-middle and constrained generation for Holo Q's verifiable coder/agent role; and **verification by diversity** — Dream shares Qwen2.5-7B's weights but decodes by a different mechanism, so an AR derivation and a diffusion derivation of the same answer cross-check each other. Above all it proves the substrate's central claim — *the model is content-addressed bytes and decode is just a transform* — holds beyond one decode paradigm, the decode-side analog of what hologram-ai's ONNX compiler (ADR-0017) did for formats.

**Honest costs.** The WebGPU masked-diffusion forward kernel is real GPU engineering, not a config change — until it lands, Dream does not generate inside Holo Q. On-device 7B is heavy: the validation ran at int4 RTN (≈0.12 worst-tensor relerr — visible quality loss; AWQ/GPTQ + more steps are the quality path) because the reference box (an iGPU-carved 8 GB ceiling) cannot hold 15 GB bf16. The int4→κ-disk publication pass (so the weights stream + re-derive sector-by-sector like every other Holo Q model) is owed. And the determinism guarantee is conditional on temperature 0 + a fixed schedule; a sampled run is non-reproducible by construction (honest L5-over-the-params, as everywhere).

**Composition.** Extends 0052 (Holo Q) and 0067 (Holo QVAC); reuses 0024/0025 (PROV-O/UOR receipt), 0033 (conscience), 0051 (Forge transform idiom), 0043 (Atlas isomorphism). Mints nothing new — `hosqvac:decode` and the Dream descriptor are data under the existing ontology.
