# ADR-0056: Holo Lang — language intelligence as a serverless, off-thread, content-addressed language service (the LSP pattern with no server), the path to real WASM language servers

**Status:** Accepted — witnessed: `tools/holo-lang-witness.mjs` is green (5/5 in real Chromium) and `#holo-lang` is a required row in `os/etc/conformance.jsonld` (the gate, ADR-024). **The escalation is realized:** a real production WASM language server — **Biome** (`@biomejs/wasm-web` 1.9.4, Rust→WebAssembly) — is vendored at `Hologram Apps/apps/workspace/vendor/biome/` (~14 MB, lazy-loaded) and driven in a Web Worker (`holo-biome.worker.mjs`), producing genuine lint diagnostics + formatting for JS/TS/JSON entirely in-tab; witnessed by `tools/holo-wasm-lsp-witness.mjs` (`#holo-wasm-lsp`, also required). **And the power ceiling is reached:** the full **TypeScript** language service (the real TS compiler, bundled in Monaco — the editor's #1 language server) runs in a κ-verified Web Worker (`vendor/monaco/language/typescript/tsWorker.js`, 5.7 MB, lazy), delivering genuine **cross-file type inference** (a real type error is reported) + IntelliSense (completion · hover · go-to-def · signature · rename · references) in-tab; witnessed by `tools/holo-ts-witness.mjs` (`#holo-ts`, required). Gate `tools/gate.mjs` green. Ships inside the Build pillar (ADR-0055) at `apps/workspace/index.html`; exposes `window.HoloLang.request(...)` (the analyzer) and `window.HoloBiome.request(...)` (the WASM server). Composes Holo Forge (ADR-0051, the compiler as the language's authority κ-object) and the workbench (ADR-0055).

**Context.** ADR-0055 stated the one boundary the Build pillar will not cross to stay serverless: a universal VS Code *extension host* needs Node + a server. But the single most valuable thing extensions provide — **language intelligence** (completion, diagnostics, go-to-symbol, references, rename, signature help, formatting) — is delivered in VS Code through the **Language Server Protocol**: a *separate process* (the "language server") speaks a documented wire protocol to the editor's *language client*. That separation is the unlock for us. A language server is just a deterministic function over source text; it does not inherently need a server *machine* — only a *process boundary*. In the browser that boundary is a **Web Worker**, and the server itself can be a **content-addressed κ-object** (today our Holo Forge compiler; tomorrow a real `clangd`/`rust-analyzer`/`typescript` compiled to WebAssembly). So the question "how do we get IDE-grade language features without a server or an extension host?" has a clean answer: run the language server as a κ-object in a worker and connect it to Monaco's provider API — the LSP architecture, minus the server machine. This also serves "text input must be crisp": analysis off the main thread never blocks a keystroke.

**Decision.** Ship **Holo Lang**: language intelligence as an off-thread, serverless language service.

1. **The analyzer runs in a Web Worker, no server, no bundler.** The Holo-C analyzer is authored as an ordinary function and shipped to a Worker via `Function.prototype.toString()` + a Blob URL — zero build step, same-origin, offline. It answers a tiny request/response protocol (`{id, method, …} → {id, result}`) in pure offsets, so it is position-agnostic and editor-agnostic — exactly the shape an LSP server has. Analysis therefore happens **off the main thread**; typing is never blocked.

2. **It feeds Monaco's standard provider API.** The main thread registers the documented `monaco.languages` providers — `DocumentSymbol` (outline), `Reference`, `Rename`, `SignatureHelp`, `DocumentFormatting` — each translating Monaco positions ⇄ worker offsets. Registering them lights up VS Code's native interactions for free: F2 rename, Shift+Alt+F format, Shift+F12 references, plus a Ctrl+Shift+O "Go to Symbol" picker and Command-Palette entries (ADR-0055). Diagnostics remain bound to the **compiler** (ADR-0051) — the language's authority κ-object — so an error is the *real* compiler's verdict, not a heuristic.

3. **The service is itself addressable.** `window.HoloLang.request(method, payload)` exposes the language service as a callable object — usable by the IDE, by tests, and by an AI agent the same way it uses `HoloApp` / `HoloCoedit`.

A required `#holo-lang` row + `holo-lang-witness.mjs` prove it offline in real Chromium: the worker answers; document symbols return the file's functions; find-references is complete; signature help resolves the called function's parameters; formatting re-indents — all with no server.

**Consequences.** The Build pillar now has real language intelligence while staying 100% serverless and offline — the LSP *architecture* without the LSP *server machine*. The honest scope: today's analyzer is scoped to Holo-C and is parse-based (file-scoped references; a conservative, whitespace-only formatter that cannot mangle code). That is deliberate — the **value is the transport and the provider wiring**, which are language-agnostic. The escalation is the same shape with a heavier, deliberately-pinned κ-object — and it is now **proven**: **Biome** (a production Rust→WebAssembly toolchain) is vendored as a content-addressed κ-object and run in this same Worker over this same `request` channel, giving real lint diagnostics + formatting for JS/TS/JSON, serverless and offline. It is driven exactly as Biome's own `js-api` drives it (`registerProjectFolder → openFile → pullDiagnostics → formatFile → closeFile`); the only cost is honest — a ~14 MB wasm, so it is **lazy-loaded** (fetched only on first JS/TS/JSON open) and never touches boot. The same slot took the heaviest server next — and it is **landed**: the full **TypeScript** language service (Monaco's bundled `tsWorker`, the real TS compiler) runs in a κ-verified Web Worker, giving real cross-file *type inference* + complete IntelliSense (the thing Biome cannot do — it lints/formats, TS type-checks; the two compose, like VS Code's ESLint + TS together). The flatten-vs-`vs` path quirk was the only wrinkle (our dist *is* the `vs` root, so the worker bootstrap re-applies the `vs → vendor/monaco` mapping the main thread uses). What remains is breadth — `clangd`, `rust-analyzer` for C/C++/Rust, and DAP step-debugging — each a κ-object whose identity is its bytes, dropping into this same worker. This is the serverless, more-honest substitute for "run every VS Code extension": not a marketplace of opaque trusted blobs, but language servers and tools that are content-addressed κ-objects (and, for non-language extensions, the holo-native verifiable extension model — skills/MCP, ADR-035/049). Web2 (it just works in a tab), web3 (the server is content-addressed, trustless, offline) and AI (the service is a callable object an agent drives) meet on one substrate.
