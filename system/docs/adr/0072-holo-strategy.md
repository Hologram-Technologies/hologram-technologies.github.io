# ADR-0072: Holo Strategy — the trading-strategy lifecycle (author → backtest → deploy) as content-addressed objects, by vendoring QuantConnect LEAN verbatim (Apache-2.0) as a sealed engine κ. A strategy is a self-verifying UOR object; a LEAN backtest is a *re-derivable* κ-transform with a PROV-O receipt (LEAN's determinism IS Law L5 — the Sharpe re-derives or is refused, you never trust a server's number); live deployment routes every LEAN order through Holo Trade's default-deny wallet gate as a conscience-gated, revocable UCAN agent. Humans write QCAlgorithm in C#/Python; agents drive the identical lifecycle over MCP. No engine code is written — LEAN is the engine; Hologram supplies identity, content-addressing, conscience, and proof.

**Status:** Proposed — to be witnessed by `tools/holo-strategy-witness.mjs` (pure Node, offline: the deterministic backtest re-derivation is exercised against a *pinned* fixture; the live `POST /exchange` stays mocked as in ADR-0048/0070) and a required `#holo-strategy` row in `os/etc/conformance.jsonld` (the gate, ADR-0024), with a live `#holo-strategy-backtest` row (a sealed strategy + pinned data re-derive the same result κ twice) and a `holo-strategy` row in `specs.json` (ADR-0031). The runtime is a new holospace app `Hologram Apps/apps/strategy` (the author/backtest/deploy surface, beside Holo Trade), the **vendored, sealed** LEAN engine, and a **sealed LEAN-data descriptor**; it reuses Holo Trade's Hyperliquid candle layer for history and Holo Trade's wallet seam for execution — adding no venue code. The decision is not done until it is witnessed (ADR-0024); this ADR pins the design and carries the build spec. *(House-keeping: `docs/adr/` numbering is being renumbered concurrently — `0070` is `holo-trade`, `0071` is `holo-bittensor`; `0072` was the max-free slot at writing. Re-grep before committing.)* Composes Holo Trade (ADR-0070, the Hyperliquid signer + candle data), Holo Forge (ADR-0051/0055, a build/compute is a re-derivable κ-transform), Holo Route (ADR-0069, the typed data-boundary pipeline), Holo Orchestrate (ADR-0045, the PROV-O run receipt), Holo Delegate (ADR-0042, the UCAN agent), the Holo Constitution (ADR-0033, the pre-dispatch gate), Holo Settle (ADR-0048/0068, payout), Holo Agents (ADR-0049, the MCP doors) and the UOR envelope + `verifyDeep` (ADR-0025). **Mints nothing** beyond a new `hostrat:` namespace, the sealed LEAN engine κ, and the sealed LEAN-data descriptor.

**Context.** Holo Trade (ADR-0070) gave a holospace a self-sovereign *terminal* — read the book, sign an order with your own key. The next want is a *factory*: let humans **and AI agents** create, test, and deploy trading strategies seamlessly, with every artifact anchored in the content-addressed substrate so a strategy and its track record are facts you verify, not claims you trust. The engine for that should be borrowed, not built — and **QuantConnect LEAN is the right one to borrow.** It is Apache-2.0 (cleanly vendorable, like QVAC and Lightweight Charts — and unlike the AGPL alternatives that drag network-copyleft into the substrate); it is an event-driven, professional-caliber engine (C#/.NET 9, with Python algorithms via `Algorithm.Python`); its strategy spec is a small, stable surface — a `QCAlgorithm` subclass implementing `Initialize()`, `OnData()`, `SetHoldings()`; its handlers (data feed, algorithm, brokerage, transaction, result) are modular and pluggable; and it runs the **same algorithm logic in backtest and live**.

The decisive property is the one QuantConnect built for reproducibility and we need for proof: **a LEAN backtest is deterministic.** A deterministic engine over content-addressed inputs produces a content-addressed output — which means a backtest can be a *re-derivable* object, not a number on someone's dashboard. QuantConnect's cloud asks you to trust that the Sharpe ratio it shows you came from your code on that data; the substrate can make the Sharpe *re-derive from* `(engine ⊕ strategy ⊕ data ⊕ config)`, so anyone — a counterparty, an allocator, an agent — recomputes it and gets the same κ or rejects it. That is the whole UOR thesis applied to quant research, and it is the magic here.

Two perils make this an ADR and not a patch, and both are about honesty. First, **LEAN is a heavyweight, server-class .NET engine** (designed for Docker / the Lean CLI), and the substrate is browser-peer and server-averse (Laws L1/L4). *Where* the engine executes must be stated precisely, not hand-waved into "it runs in the browser." Second, **LEAN has no Hyperliquid brokerage** — and the instruction is to write no new code. The resolution is that Holo Trade *already is* the Hyperliquid signer: for backtests no brokerage exists at all, and for live trading LEAN's order intents route through Holo Trade's wallet-gated seam rather than a new brokerage plugin. We adhere strictly to LEAN's source and specs; we add only the substrate around them.

**Decision.** Make the strategy lifecycle three content-addressed objects over a vendored LEAN, writing no engine code and adding exactly four things:

1. **LEAN is vendored verbatim and *sealed* — Hologram writes zero engine code (NIHITO, Law L4).** The QuantConnect LEAN engine (Apache-2.0) is vendored and pinned by a self-verifying `os/usr/lib/holo/lean-engine/lean-engine.uor.json` (ADR-0025): its κ commits to every shipped byte (RFC 8785 / SRI), so a tampered engine changes the κ and is **refused before it runs** (Law L5). The strategy-authoring surface *is* LEAN's `QCAlgorithm` API verbatim — `Initialize` / `OnData` / `SetHoldings`, C# or Python — with no Hologram dialect layered on top. We strictly adhere to LEAN's source and the LEAN data format; the attribution and NOTICE are preserved per Apache-2.0.

2. **A strategy is a sealed UOR object (Law L1).** An algorithm (its `QCAlgorithm` source file(s) ⊕ its run config ⊕ the engine κ it targets) is sealed as a `hostrat:Strategy` UOR object whose κ is its identity; an edit advances the κ. The object pins the *exact* algorithm bytes, so "strategy X" is a content address, not a name on a host — two people referencing the same κ have provably the same code.

3. **A backtest is a re-derivable κ-transform with a PROV-O receipt (Law L5) — the keystone.** Running the sealed LEAN engine κ over **content-addressed historical data** (the LEAN data format, transcoded from the Hyperliquid `candleSnapshot` history Holo Trade already streams, pinned by κ) and a pinned config yields a result that is a *pure function* of `(engineκ ⊕ strategyκ ⊕ dataκ ⊕ configκ)`. Because LEAN backtests are deterministic, anyone re-runs and re-derives the **same result κ** — the run is a `hostrat:Backtest` PROV-O entity (`prov:used` the engine/strategy/data, `prov:wasGeneratedBy` the run activity, the statistics and equity curve in its body), and a track record is a chain of such κ. **No trusted server decides your Sharpe — it re-derives or it is refused.** This is the Forge thesis (ADR-0051: a build is a re-derivable κ-transform) applied to a simulation, over the Route data boundary (ADR-0069), and it is *strictly better* than the cloud original: there you trust the result; here you verify it. Memoization falls out for free (Law L3): the composite key memoizes the result, so re-running an unchanged `(engine ⊕ strategy ⊕ data ⊕ config)` is an O(1) κ-fetch.

4. **Live deployment routes every LEAN order through Holo Trade's default-deny wallet gate as a conscience-gated, revocable UCAN agent (Law L4 + ADR-0070/0042/0033) — no new brokerage is written.** LEAN's live mode emits order intents; instead of a Hyperliquid brokerage plugin, the intents are dispatched through Holo Trade's signing seam (ADR-0070): the master key never enters the engine, routine orders are signed by the `approveAgent` withdrawal-less key wrapped as a **revocable, attenuated Holo Delegate UCAN** (ADR-0042), every order passes the Constitution's pre-dispatch conscience gate (ADR-0033, the strategy's own risk limits), and every fill seals as a `hostrade:Trade` receipt. A running strategy is a **Holo Orchestrate execution DAG** (ADR-0045) whose root κ proves the whole session; P&L settles through Holo Settle (ADR-0048/0068). Deploy is opt-in, revocable in one act (kill the UCAN, re-key the agent), and provable end-to-end.

**Where LEAN runs — the honest split (one UOR guarantee, two loci).** This is stated precisely because the Laws demand it:
- **WASM-native — the Law L1/L4 ideal.** LEAN compiled to WebAssembly via the .NET `wasm` workload and sealed as a browser-resident engine κ that runs the backtest in-tab, no server. This is the **target**, and it is a substantial build: LEAN is large and .NET-on-WASM has real threading/IO constraints. **We do not claim it runs in-browser today.** The ADR commits to the build; the witness proves the *seal and the re-derivation*; the `#holo-strategy-backtest` render row lands when the WASM engine boots.
- **Operator-opt-in local runner — the honest interim.** The official Lean CLI / `quantconnect/lean` Docker image, run by the sovereign user on their own machine — **never core infrastructure**, exactly the Plasma operator-opt-in boundary (ADR-0068, Law L4). The substrate wraps its inputs (`engineκ ⊕ strategyκ ⊕ dataκ ⊕ configκ`) and its output (`resultκ` + receipt) so the invariant — *re-derivability* — holds regardless of where the deterministic engine executed. The execution locus is an implementation detail; the content-addressed proof is the thing that matters, and it is identical either way.

**Magical — for the quant, and for the agent that earns its own track record.** The lifecycle is three verbs over objects you already trust, and a human and an agent use the *same* ones:

```js
import { HoloStrategy } from "@hologram/strategy";   // this ADR — wraps the sealed LEAN engine κ
import { hyperliquid } from "@hologram/chains";       // Holo Trade's sealed venue + candle history
import { wallet } from "@hologram/wallet";            // the default-deny signing seam (ADR-0070)

// 1. author — a QCAlgorithm (C#/Python), sealed to a κ that IS its identity (Law L1)
const strat = await HoloStrategy.seal({ algorithm: "MyMomentum.py", config });

// 2. backtest — deterministic ⇒ the result re-derives; anyone recomputes the same κ (Law L5)
const bt = await HoloStrategy.backtest(strat, { data: hyperliquid.history("ETH", "1h"), engine });
//   bt.receiptκ → engineκ ⊕ stratκ ⊕ dataκ ⊕ configκ ⊕ stats. Pin it, share it, an allocator VERIFIES it.

// 3. deploy — live, through Holo Trade's gate: conscience-checked, revocable UCAN, every fill a receipt
const run = await HoloStrategy.deploy(strat, { signer: wallet, venue: hyperliquid, ttl: "24h",
                                               maxNotional: 25_000 });   // Orchestrate DAG + Settle payout
await run.revoke();   // one act ends it; the master key never moved
```

An **agent** gets the identical lifecycle over MCP (ADR-0049): `author_strategy` (seal a QCAlgorithm → κ), `backtest_strategy` (re-derive a result κ + receipt), `verify_backtest` (re-derive any *peer's* claimed result — un-fakeable track records), `deploy_strategy` (a conscience-gated UCAN over Holo Trade). Because an agent's backtest is a *re-derivable* object, an agent can build a **verifiable** quant reputation (composing Holo AgentTrust, ADR-0039) — and get paid against proven live performance (Holo Settle). A human hands an agent a strategy κ, the agent improves it and returns a new κ with a re-derivable backtest, and neither has to trust the other's machine. Create → test → deploy, seamless, by humans and agents, every step a fact.

**Relationship to what exists (the boundary that is actually new).** This ADR introduces *no* new economic or execution primitives — it is composition. The history is Holo Trade's candle layer; the signer, key custody, agent-wallet UCAN and conscience gate are Holo Trade's (unmodified); the "deterministic compute → content-addressed result with a receipt" machinery is Forge's and Route's; the run DAG is Orchestrate's; the payout is Settle's; the agent doors are Holo Agents'. The genuinely new objects are three, and their novelty is the point: the **sealed LEAN engine κ** (so the engine our proofs depend on is a content-addressed fact, not a Docker tag that drifts), the **`hostrat:` vocabulary** (binding strategy/backtest to schema.org + PROV-O), and the **sealed LEAN-data descriptor** (so the historical data a backtest consumes is pinned, not fetched-and-trusted). It mirrors Holo Trade exactly — *vendor the official engine, seal it, wrap it with identity, conscience, and proof* — one tier up the stack.

**Consequences.** Humans and agents get a content-addressed strategy factory over Holo Trade: a strategy is a κ, a backtest is a re-derivable proof with a receipt, a deployment is a conscience-gated revocable UCAN whose every fill is checkable, and a track record is a chain of κ no one can forge. The honest scope and costs: (a) **backtest determinism is load-bearing** — LEAN is deterministic by design, but version, config, data, and (for Python) the interpreter must be pinned; the engine κ + data descriptor + config κ do exactly that, and the witness *proves* twice-equal re-derivation rather than asserting it. (b) **WASM-native is a real, unfinished build** — the interim is the operator-opt-in local Lean runner, and we make no false claim of in-browser execution today; either locus yields the same re-derivable proof. (c) **live execution still custodies margin on Hyperliquid** — Holo Strategy inherits Holo Trade's honest boundary (we run the lifecycle and the proofs, not the venue's matching or its USDC bridge). (d) **no Hyperliquid LEAN brokerage is written** — live orders route through Holo Trade's seam, so the "no new engine/brokerage code" rule holds; a strategy can also be deployed against LEAN's *own* supported crypto brokerages by the operator, opt-in. (e) **Apache-2.0** keeps vendoring clean (attribution/NOTICE preserved). (f) **Quantum honesty, inherited:** the strategy, the backtest, and the receipt are hash-based content commitments ⇒ post-quantum (independent of the venue's classical secp256k1 signature and finality, which we do not claim to change). It composes the catalog rather than extending it — minting nothing but one vocabulary, the sealed engine, and one sealed data descriptor.
