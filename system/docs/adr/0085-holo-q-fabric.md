# ADR-0085: Holo Q Fabric — content-addressed compute for the specialist fleet: every model call is κ-memoized (O(1) replay), streamed, and κ-addressed for render reuse — the layer that converges a universe of small models toward a real-time, LLM-like rendered experience

**Status:** Accepted (fabric + the explicit Holo Mind orchestrator binding + the κ-addressed renderer landed + Node-witnessed — 11/11 fabric, 12/12 end-to-end, 8/8 renderer — and shown LIVE in a creator-mode demo; the persisted index + real model planning pending). The κ-addressed renderer ships at `holo-q-render.js` (structured-DOM only — never raw innerHTML — so model output can't inject script; a witness caught and fixed a style-allowlist hole that permitted `url(javascript:)`); it paints the fabric's streamed output live and reuses identical fragments by content κ in O(1). A self-contained creator-mode showcase (`studio-demo.html`) wires prompt → fabric → renderer and was verified live in the browser preview: a prompt builds a hero in the preview pane, **0.88 ms cold**, and a repeat is a **cache hit + render-reuse at 0.07 ms** (BOTH O(1) layers), with **0 network calls** (100% serverless — a `fetch` guard counts them on screen). Ships at `system/os/usr/lib/holo/q/holo-q-fabric.js` (buildless, dependency-free, DOM-free); the binding at `holo-q-orchestrate.js` reuses the REAL orchestrator core (`holo-mind.mjs` `runLoop`: intent → plan → gate → dispatch → seal) with its dispatch seam wired to the fabric — the helper tasks become roster verbs (Law L4), the conscience gates each, and the action receipt's `effectKappa` IS the fabric's content κ. Proven end-to-end (12/12): the loop plans → gates → dispatches a specialist through the fabric and seals a receipt that **re-derives byte-for-byte** (Law L5); a repeat ask hits BOTH O(1) layers (the plan memo and the fabric compute memo — **1.24 ms vs 32.7 ms cold**, the specialist runs once); the entire loop makes **zero network calls** (100% serverless — `fetch` stubbed to throw, never hit); and a conscience-blocked verb dispatches and seals nothing. It generalizes the substrate's ONE content-addressed-compute idiom — Holo Q's answer memo (ADR-0052), Holo Mind's plan memo (`memoKey`, ADR-0081), the κ-store (`holo-store.js`, Law L3/L4) — across EVERY Mixture-of-Specialists call (ADR-0084): the first time an `(model, input, params)` is seen it computes/streams once and seals the output to a content κ; every identical ask after returns in **O(1)** with no compute and no token stream. One streaming `run()` interface fits both instant specialists (embeddings) and generative ones (live deltas → a sealed final), and every event carries its content κ so a κ-addressed renderer reuses identical on-screen fragments. Validated in Node 11/11: cold compute seals a κ; an identical ask replays O(1) hot with the SAME κ and the provider runs exactly once; distinct input/params → distinct κ; a generative provider streams deltas then a sealed final, and a replay is one instant final (no re-stream); outputs seal into the κ-store (durable); identical output → identical render key. **Deferred:** mounting the renderer into the REAL Create studio (the showcase proves the loop in a self-contained page; binding it to the shell's build flow is the next step) and a WebGPU/canvas node path beside the DOM one, persisting the input→κ index so O(1) survives a reload (sealed bytes already do), real model planning over the binding (the QVAC sampler in place of the serverless rule-planner floor) and the `reevaluate` self-improvement policy on a schedule, and a `#holo-q-fabric` witness row + os-closure re-lock.

**Context.** ADR-0084 gave the OS a fleet of tiny, auto-discovered specialists and ADR-0084's first adapter proved one runs on-device in ~6 ms. But a *fleet* of models is only "real-time" — only able to feed content that renders and updates live on screen — if repeated and structural work is free. Two facts make that reachable on this substrate and nowhere else: a model output is bytes, so it is a content-addressed κ-object; and a specialist call at fixed params is a deterministic transform, so the SAME `(model, input, params)` always yields the SAME output. Together they mean an answer need be computed *once* in the life of the system: address it by content, and every later identical ask is a lookup, not a computation. The substrate already does this in three narrow places. What was missing is the layer that makes it the DEFAULT for the whole specialist surface — so that as the orchestrator fans work across many micro-models, the hot path collapses to memory speed and the cold path streams, which is exactly the shape of a fast LLM UI.

**Decision.** Ship a compute **fabric** every specialist call rides, built on the existing rails, not beside them:

1. **One memo, content-addressed (Law L3).** `run()` computes `κ(model ⊕ input ⊕ params)` (the `holo-mind.memoKey` idiom generalized) and checks a hot in-memory LRU, then a durable index into the OS κ-store. A hit returns the sealed output with no compute. A miss computes once, seals the output to its content κ (`store.put(bytes) → κ`, re-derivable, Law L2), and records it. Identical work is never repeated — the core latency win, applied universally.
2. **One streaming interface for instant AND generative.** `run()` is a single async-iterable. An instant specialist (an embedder) yields one `final` event; a generative one yields live `delta` events (append for autoregressive, block-refine for diffusion — ADR-0083) and then a sealed `final`. A cached ask of EITHER kind yields one `final` in O(1) with no stream. This is the streaming-LLM experience as a substrate primitive, not a per-app feature.
3. **κ-addressed rendering (the convergence hook).** Every result carries its content κ, and `renderKey(value)` returns it, so a UI keeps `Map<κ, node>` and skips re-rendering identical output — compute once, address it, render once, replay. This is the honest bridge to "real-time on-screen rendered content": the fabric makes the *compute* and its *identity* cheap; the renderer that consumes the key is the next layer (widgets/`holo-gpu.js`, ADR-0061/0080).
4. **Orchestrator-driven.** Holo Mind (ADR-0081) discovers specialists (the mux), applies them through the fabric, and self-improves (`reevaluate`), with each output a re-derivable, conscience-gated receipt. The fabric is what makes its fan-out cheap enough to feel live.

**The convergence thesis (honest).** A real-time, on-screen, app-like experience streamed from a universe of small models is the trajectory, not a claim landed here. What IS landed and proven is the mechanism that unlocks it: universal content-addressed compute so the hot path is memory-speed, a uniform streaming surface so the cold path is incremental, and a content-κ on every output so rendering can be just as cheap. The remaining distance is the renderer that turns those keys into reused DOM/WebGPU nodes, and the orchestrator policy that keeps the right specialists warm — both downstream of, and enabled by, this layer.

**Honest costs.** The input→κ index is in-memory in v1, so cross-reload O(1) needs it persisted too (the sealed *bytes* are already durable — a small refinement, flagged in `describeFabric`). The κ-addressed renderer is NOT in this module — the fabric hands the UI a key; reusing nodes by it is the widgets/GPU layer's job. Determinism (hence the memo and the receipt) holds at fixed params / temperature 0; a sampled generative call is non-reproducible by construction and should bypass the seal (honest L5-over-the-params). And a memo is only as fresh as its inputs — content that depends on time or external state must fold that into the key or it will serve stale.

**Composition.** Generalizes 0052 (answer memo) and 0081 (`memoKey`) over 0084 (the specialist fleet); built on `holo-store.js` (Law L3/L4 κ-store) and the decode-agnostic receipt (0024/0025/0083); rendered by the widgets/GPU layer (0061/0080) downstream. Mints nothing — the fabric is the memo idiom as a reusable seam.
