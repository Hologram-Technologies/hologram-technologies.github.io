# ADR-0068: Holo Settle · the Plasma rail — verifiable settlement pays out in real, gasless USD₮, bound by content address to the proven work it pays for; the on-chain dollar Merkle-commits to the receipt, the entitlement stays post-quantum, and no key custody is minted

**Status:** **Accepted** — witnessed. The runtime ships in the agent-stack tree (the `hologram-os` runtime repo, where Holo Settle/Orchestrate run) as `os/holo-settle-plasma.mjs` (`PLASMA` · `transferIntent` · `redeemOnChain`) over a **sealed chain descriptor** `os/etc/holo-chains/plasma.uor.json` (re-derives to `did:holo:sha256:7fe71e7f…`), routing every actual value movement through the **already-built** Holo Wallet signing seam `os/_shared/holo-wallet-bridge.js` — default-deny, human-approval, no new crypto. It is witnessed green by `os/settle-plasma-witness.mjs` (7/7: descriptor-sealed/anti-phishing · pay-only-proven-work · memo == voucher κ · entitlement-post-quantum · default-deny + at-most-once · conservation · mint-nothing) and registered as the `holo-settle-plasma` row in `os/specs.json`, run by the gate `enforce-witnesses.mjs` (ADR-0024). The agent surface is live: a **`settle_onchain` MCP verb** in the agent roster (`os/mcp/holo-agent-mcp.mjs`) — discoverable in `/.well-known/mcp.json` + `/.well-known/agents.json` and witnessed by `agents-mcp-witness.mjs` (now **7** agent verbs, green) — that re-derives first and returns proven, gated USD₮ transfer intents (it *moves nothing*; the human wallet broadcasts). *(Repo note: this ADR doc lives in the OS2 ADR series (next free `0068`); its code lives in the `hologram-os` runtime where the agent stack actually runs — the two repos' ADR numbering has diverged, the same doc/code split the README notes for ADRs 0001–0021.)* Composes Holo Settle (ADR-0048), Holo Own's anchor-by-reference (ADR-0053), the Constitution's pre-dispatch gate (ADR-0033) and the UCAN authorization layer (ADR-0042). **Mints nothing** beyond the existing `hosc:` namespace and one sealed network descriptor.

**Context.** ADR-0048 (Holo Settle) built the hard half: it conditions payment release on the *re-derivation of a work receipt* — pay against **proven** work, never claimed work. A payer-signed `schema:Order` Merkle-commits to a work receipt and a split; `settle` releases one `schema:Invoice` voucher per contributing agent **only if** the receipt re-derives (Law L5) ∧ every step was authorized (ADR-0042) ∧ no step was conscience-blocked (ADR-0033); the voucher's κ *is* the content-addressed txId (idempotent — no double-spend on the same work); the payee redeems trustlessly. It is the capstone of the agent stack and it is witnessed green.

But ADR-0048 closed with an honest gap, in its own words: *"the voucher models the release; in production it redeems against a real rail — NANDA Points (x402-NP), Lightning (L402), or an on-chain transfer through Holo Wallet's default-deny, human-approval signing seam — keyed on the voucher κ. … only the rail is mocked offline."* The voucher's currency is `NP` (NANDA Points) — a symbolic unit. **No real money moves.** An agent that verifiably did the work is owed *something*, and that something is presently a receipt for a receipt.

Two facts make this the moment to close it. First, **the seam already exists**: `holo-wallet-bridge.js` lets any holospace ask the running Holo Wallet to `requestSend(chain, to, amount, { token })` over a `BroadcastChannel`, gated by the same human-consent dialog a person sees, default-deny (no wallet open / user declines ⇒ the request errors — a holospace can *never* move value on its own). The wallet's engine is Tether's WDK (`os/usr/lib/holo/wdk/…`), which is already EVM. Second, **Plasma is the right rail and asks us to write no crypto.** Plasma is a purpose-built Layer-1 for stablecoin payments: EVM-compatible (so the WDK drives it as a network config, not a new chain implementation — NIHITO) and, decisively, **gas-abstracted** — it sponsors zero-fee USD₮ transfers. Gas abstraction is not a nicety here; it is the unlock. The reason the `NP` voucher stayed symbolic is that a real agent settlement is a *micro*-payment (a sub-cent share of a 3-step collaboration), and on any gas-metered chain the fee eats the payment. On Plasma a 0.003 USD₮ payout costs the payee nothing in gas. **Gasless stablecoin transfer is precisely what turns "proven work" into "paid in dollars" at agent scale.**

The peril is equally precise, and it is why this is an ADR and not a patch. Plasma is an external, permissioned chain with its own compliance frameworks and a custodial consumer product; trusting its ledger blindly would violate the substrate's whole posture (Law L1 / private-first; "verify, don't trust"). And a chain config is a phishing surface — swap the RPC URL or the USD₮ contract address and you have redirected every payout. The integration must therefore (a) keep the **entitlement** — the proof that this payee is owed this amount — entirely inside the content-addressed substrate, never delegated to Plasma; (b) make the **chain parameters themselves** content-addressed and tamper-evident; and (c) move value only through the existing default-deny human gate. Do that and Plasma becomes what Holo Own already proved a chain should be: an **anchor referenced by content**, never a dependency minted into the core (Law L4).

**Decision.** Give Holo Settle a real settlement rail by adding exactly three things and re-implementing nothing:

1. **Plasma is a *sealed* chain descriptor, not a hardcoded config — so the rail itself is content-addressed and un-swappable.** `os/etc/holo-chains/plasma.uor.json` is a self-verifying UOR object (ADR-0025) pinning the chain's parameters verbatim from Plasma's official network-configuration page: `caip2: "eip155:9745"` · `chainId: 9745` (`0x2611`) · `nativeCurrency: XPL` · `rpc: "https://rpc.plasma.to"` · `explorer: "https://plasmascan.to"` · the **USD₮0 ERC-20** contract address (to be pinned *verbatim* from the official mainnet-details page — candidate `0xb8ce59fc3717ada4c02eadf9682a9e934f625ebb` — verified against that page before sealing) with its `decimals`. The descriptor's κ commits to every one of those bytes (RFC 8785 canonical form), so a swapped RPC or a substituted token address **changes the κ and is refused** — the anti-phishing guarantee is structural, not vigilance. The descriptor is registered as one entry in the WDK/wallet chain registry; because Plasma is EVM, the wallet drives it through its existing EVM account path. **No new signing, key-derivation, or transaction-construction code is written** — the BIP-39 seed identity (the Holo Login keystone) already derives the EVM key; Plasma is a network the existing account speaks to. NIHITO, Law L4 (anchor by reference; mint no pillar).

2. **The voucher gains an on-chain redemption that re-derives *before* it can move a cent (Law L5), through the default-deny human gate (Law L4).** A new pure-ish function `redeemOnChain(store, voucher, { chain = plasmaκ })` does, in order: re-derive the voucher (`verify`) and its order (`verifyProof`) and the work receipt (`verifyDeep`) — **identical to the existing `redeem` precondition, reused verbatim**; resolve the payee's EVM payout address from its `did:holo`/NANDA identity κ; convert the split share to USD₮ minor units against the descriptor's `decimals`; and **only if all of that holds**, hand a fully-formed transfer *intent* `{ chain: "plasma", to: payee, amount, token: USD₮0 }` to `requestSend` on the Holo Wallet bridge. The human approves (or declines) in the one consent dialog; the wallet builds, signs and broadcasts the gasless USD₮ transfer; the returned `txHash` is recorded back onto the voucher as `schema:PaymentMethod`/`prov:wasGeneratedBy`. **The voucher κ remains the canonical, content-addressed payment id; the Plasma `txHash` is merely the on-chain receipt bound to it.** Tampered or unproven work produces *no intent at all* — the bridge is never called, so nothing can be approved by mistake. No wallet open ⇒ default-deny ⇒ no payout. Settlement entitlement lives in the substrate; Plasma only *transports* an already-proven obligation.

3. **The two ledgers are bound by content address — the on-chain dollar Merkle-commits to the proven-work object, and the binding is post-quantum and at-most-once.** The USD₮ transfer carries the **voucher κ (a sha-256 content address)** as a content-addressed memo — in the transfer's calldata / an event log the descriptor names. So the dollar movement on Plasma *references the exact receipt-of-proven-work* it settles: anyone holding the `txHash` re-derives the voucher → the order → the receipt and confirms the on-chain payment corresponds to a real, authorized, conscience-accepted collaboration — **the open-web audit trail Holo AgentTrust and Holo Orchestrate were built to make checkable, now reaching all the way to the money.** Because `(receipt ⊕ order) → voucher κ` is deterministic and idempotent (ADR-0048), the memo is stable, and `redeemOnChain` keys an **at-most-once** guard on the voucher κ (a κ already bound to a `txHash` will not re-send) — the on-chain analogue of "no double-spend on the same work." This is where the UOR substrate makes the rail *more* secure than the rail: the **entitlement** — the binding "this payee is owed this amount for this work" — is a hash-based content commitment, which is **quantum-resistant**; it does not rest on Plasma's (classical secp256k1 / classical-consensus) finality. We are honest about the split: Plasma's *settlement finality* is classical and we do not claim otherwise; what the substrate guarantees post-quantum is the *entitlement and the audit trail*, which is exactly the part you must not be able to forge. A future post-quantum signature on the order/voucher (the substrate's `holo-vc` seam is swappable) upgrades the authorization layer without touching this design.

**Magical for builders — and for the guest who watches an agent get paid.** The surface does not grow a new product; it grows one clause on the verb that already exists:

```js
import { settle, redeemOnChain } from "@hologram/settle";   // ADR-0048 + this ADR
import { plasma } from "@hologram/chains";                   // the sealed descriptor κ

// 1. prove + release (unchanged from ADR-0048): vouchers exist ONLY if the work re-derives
const { vouchers } = settle(store, receipt, order);

// 2. pay for real — re-derives first, then asks the wallet (you approve once), gasless USD₮ on Plasma
for (const v of vouchers) {
  const { txHash } = await redeemOnChain(store, v, { chain: plasma });   // ← the only new line
  // txHash is bound to v's κ; the on-chain transfer's memo === v's κ === the proof-of-work
}
```

No SDK to install, no server, no new keys. An **agent** gets the same power over MCP: a `settle_onchain` tool re-derives the voucher and returns the signed transfer intent for the wallet to gate — verify, don't trust. And the human experience is the magic ADR-0064 is chasing: a guest remixes something, an agent does a slice of verifiable work, and **the instant that work re-derives the guest watches it get paid real dollars** — zero gas, 24/7, no intermediary, with a clickable receipt that ties the dollar to the exact computation. Proof-of-work → dollars, with nothing in between to trust.

**Relationship to what exists (the boundary that is actually new).** This ADR adds *zero* new economic primitives — it is the production wiring ADR-0048 explicitly deferred. `settle`/`redeem` and the split, the order, the conditions are untouched; `redeemOnChain` reuses `redeem`'s re-derivation preconditions verbatim and only appends the bridge call. It mirrors **Holo Own (ADR-0053)** exactly: Own anchors *scarcity* to an existing chain "by reference … only when an asset needs it, never minted (Law L4)"; this anchors *settlement* to Plasma by reference, only when a payout needs a real rail, never minting custody. The wallet seam (`holo-wallet-bridge.js`) and the WDK are pre-existing and unmodified — Plasma is a registry entry. The one genuinely new object is the **sealed chain descriptor**, and its novelty is the point: it makes the rail's parameters a content-addressed, tamper-evident fact rather than a config string an attacker can edit.

**Consequences.** The trustless agent economy stops being symbolic: a voucher that today says "100 NP" can pay out real, gasless USD₮ on Plasma, released only against work that re-derives, moved only through a human-approved default-deny gate, and bound by content address to the proof it settles. Three properties hold that a raw on-chain transfer cannot offer: **entitlement is post-quantum** (hash-based, independent of the chain's classical finality), **the dollar is auditable to the computation** (the on-chain memo is the voucher κ is the receipt), and **the rail is un-phishable** (the chain parameters are sealed; a swapped RPC/token changes the κ and is refused). The honest scope and costs: (a) we pin **Plasma mainnet beta** parameters — they must be copied verbatim from the official network-configuration page and *re-verified* before sealing (a wrong USD₮ address sealed is a wrong address made tamper-*evident*, not correct — the human must check once); (b) the witness proves the *intent construction and binding* offline (voucher re-derives ⇒ correct EVM transfer intent to the right `to`/`amount`/`token`/`chainId 9745`; tampered work ⇒ **no** intent; memo === voucher κ; idempotence ⇒ at-most-once; no wallet ⇒ default-deny), while the *broadcast* itself is exercised only against the live chain (the rail stays mocked in the gate, as in ADR-0048); (c) Plasma is permissioned with compliance frameworks — it is an **operator-opt-in** rail, never a core dependency and never a gate requirement, the same boundary the substrate drew rejecting hosted/gated stacks; (d) a payee needs an EVM payout address resolvable from its identity κ — agents without one simply keep the symbolic `NP` voucher (the NP and USD₮ rails coexist; the voucher is rail-agnostic). It composes the catalog rather than extending it: Settle's conditions, Own's anchor-by-reference, the Constitution's gate, Delegate's authorization, and the Wallet's signing seam — mint nothing new but one sealed descriptor.
